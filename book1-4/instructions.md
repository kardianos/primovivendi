# Project Instructions for Writing Chapters

Create the complete book by writing one chapter per file in the `chapters/` directory. A simple concatenation of the files in lexical order by filename produces the full manuscript ready for pandoc. Follow every rule below without exception.

## File Naming Convention

Name every content file with a leading sequence number of the form `<00part><00chapter>_<slug>.md`.

- `00part` is the two-digit part number starting at 01.
- `00chapter` is the two-digit chapter number within its part, starting at 01 for the first chapter of that part.
- `slug` is a short, lowercase, underscore-separated identifier that reflects the chapter title.

The title block file is the single exception: name it `0000_title_block.md`.

Examples of valid names:
- `chapters/0000_title_block.md`
- `chapters/0101_introduction.md`
- `chapters/0302_self.md`
- `chapters/0405_philosophical_landscape.md`
- `chapters/0507_live_well.md`

Sort the directory by filename and concatenate in that order to join the book. Never change the sequence numbers after files exist.

## Part Titles by Function

Name each part according to the function it performs in presenting the concepts. Never use the word "ring", "seed", "unfold", or any other structural label that refers to the book architecture itself.

The five parts and their titles are:

- Part 01: The Five Principles
- Part 02: Foundations
- Part 03: Reality, Self, and Others
- Part 04: Ramifications, Contrasts, and Context
- Part 05: Application and Practice

Use these exact titles in the `#` part headers described below.

## Title Block File

The file `0000_title_block.md` contains only a pandoc-compatible YAML frontmatter block and nothing else. It must be the first file in the concatenation. No later chapter file may contain a YAML block delimited by `---`.

Use this structure (a single pandoc YAML block). Declare variables under a custom key such as `substitutions`:

```
---
title: "The Conservative Frame"
author: "Author Name"
substitutions:
  book_title: "The Conservative Frame"
  five_principles: "the five principles"
  bundle: "the bundle"
  procedural_symmetry: "procedural symmetry"
---
```

Declare any title or concept name that might still change before publication under the `vars` key. In chapter content, refer to the mutable name with the syntax `{{.variable_name}}`. A pre-processor will perform the substitution before the joined file reaches pandoc. Use this mechanism for the book title itself and for any principle or section name still under review.

## Header Levels and File Structure

- Place the `# Part XX: Title` line only in the first chapter file of each part.
- Place the `## Chapter Title` line at the very beginning of every chapter file (after the part header when the file also carries the part header).
- Use `###` for every section header inside a chapter.
- Never place a `#` header anywhere except as the part header in the opening file of a part.
- Never place a `##` header anywhere except as the chapter title at the start of its file.
- The joined manuscript therefore contains part headers, chapter titles, and section headers in the correct hierarchy without any file needing to know the global structure.

## Writing Style Rules

Write every sentence in the chapter files literally and declaratively. State exactly what the concepts require. Do not use allegory, metaphor, or illustrative parables. Forbidden example: "You are a shipbuilder in life and life is the road you travel." Required example: "As time passes during a human life, people make many decisions about how to shape their actions and their environment."

Connect ideas with explicit connecting tissue between concepts. Most readers prefer concepts to be introduced gradually rather than packed densely into few sentences. Spread key distinctions, definitions, and relationships across multiple sentences and short paragraphs. Bring the reader into each idea through context and natural transition before moving to the next. Preserve every required concept from the chapter plan, but reduce concept load density so the prose feels readable rather than compressed.

Do not describe the book structure inside the chapter prose. Do not say "this chapter develops," "in this part," "as the next chapter shows," "we have already seen," or any similar meta-commentary. The sequence of files and the headers carry the structure; the sentences present only the concepts themselves.

Never use the word "ring" or any concentric-circle language in any chapter content.

Punctuation is limited to the following marks: . , ; : ( ) " ' ? ! together with the emphasis markers * and ** when required. Place exactly one space after every period. Do not use em-dashes or en-dashes under any circumstances. Use a semicolon, a colon, or parentheses to separate related clauses instead.

You may use **bold** and *italic* for emphasis when doing so increases clarity. You may use tables, numbered lists, and bullet outlines when they present relationships or procedures more clearly than prose alone.

Every sentence must contain useful, defensible content about the concepts. No sentence may exist only to fill space or to create a transition that comments on the writing itself.

## Content Development Process

1. Read the detailed outline in `outline/toc_r05.md` and the chapter plan in `outline/toc_r05_chapters.md` for the exact material assigned to each file.
2. Write the assigned concepts in the order they appear in the outline, using only the style rules above.
3. When a concept name or the book title remains provisional, declare the current best string in the `vars` section of `0000_title_block.md` and reference it with `{{.variable_name}}` in the chapter text.
4. Keep each chapter file focused on its assigned material. Do not move content between chapters.
5. After writing or revising a file, verify that it begins with the correct header(s), uses only allowed punctuation, contains no structural meta-talk, and states every claim literally.

## Response Protocol When Writing Chapter Files

When creating or updating any chapter file, write the file directly using the `write` tool. Do not repeat or paste the full chapter content back into the response to the user. This saves tokens and keeps the conversation focused on review.

In the response after writing, state only the following:
- The exact filename that was created or modified.
- If any adjustments were made to an existing file, briefly list the specific concepts, sections, or wording that were updated.

Do not re-explain the chapter's content or quote large portions of it unless the user specifically requests the current text of a file.

These instructions are imperative. Follow them exactly when creating or editing any chapter file.
